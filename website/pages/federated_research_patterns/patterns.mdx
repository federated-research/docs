# Computational Patterns
There are dozens of ways to undertake federated analysis or federated learning and there are benefits and limitations of those processes.

Our work has shown that despite the significant number of federated tools, algprithms and platforms, when you examine 
the patterns for how these analyses are undertaken they can be simply grouped based on two criteria, 

| Component         | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              | 
|-------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Analytical**    | There are three key concpets that summarise the vast number of analytics and learning that are possible when data is spread across multiple TREs:<br/><br/> **Isolated** – these are analyses that are replicated individually within each TRE. <br/><br/>**Connected** – these are analyses which require multiple rounds of local calculation and aggregation, requiring the TREs to receive results from other TREs to be included in the local calculations. <br/><br/>**Centralised** – these analyses require data to be pooled, even temporarily, for the analyses to be performed.                                               |
| **Data Movement** | As deteremined by the algorithm type, this is how data is required to moves in and out of a TRE and how it is exectued. <br/><br/>**Summary:** where a statistical result is shared back that does not relate to any individual but is a calculation based on the confidential data. It is the summary result that moves from the TRE. <br/><br/>**Model Parameters:** where a machine learning algorithm has been run on the data and the weights are moved from the TRE. <br/><br/>**Row level data**:where row level data is moving, whether that is anonymised (or not) but each data element is that at the level of an individual. |


## Connections

| Analytical | Data Movement | 
|-|-|
| **Isolated** | Summary only| 
| **Connected** | Model Parameters or Row level data| 
| **Centralised**| Row level data only| 

## Common Patterns


### Traditional Pattern
**Analytical:** Isolated <br/>
**Data Movement**: Summary <br/>
**Required Trust between TREs**: Low <br/>

The most common method to undertake federated analytics is for the analyses to be performed in each location individually, summary results are gathered centrally and further meta-style analyses are performed. This pattern requires the lowest trust between TREs, as each result is undertaken independently and a meta or two stage analysis is performed over the results. 

The limitation of this approach is that data cannot be linked and certain analyses cannot be undertaken via this method. Therefore, the most traditional approach, but the least powerful for a researcher.

### Learning Pattern
**Analytical:** Connected <br/>
**Data Movement**: Model Parameters <br/>
**Required Trust between TREs**: High <br/>

The most commons method to undertake federated learning is for the model parameters to be passed via a central node, or directly between the TREs. The key difference for a TRE is that results are combined and shared between other TREs, therefore there is a higher trust required between the TREs, as they must trust the sharing of model parameters with the researcher and the TREs. 
![Alt Text](/assets/gif/Connected%20Analytics.gif)


### Dynamic Pooling Pattern
**Analytical:** Centralised <br/>
**Data Movement**: Row level data <br/>
**Required Trust between TREs**: Very High <br/>

An emerging pattern from the DARE UK TREvolution programme is that a project specific network is created between the TREs, to either move or provide the researcher with a view of the data that makes it appear as if the data is all in one location. This approach is by far the most powerful for a user and the analyses and learning are almost unlimited.

This approach does require a very high level of trust between TREs as it would require effective row level sharing of data between TREs. Disclosure control would still be applied before the data left the TRE network.
