# Computational Patterns
There are dozens of ways to undertake federated analysis or federated learning and there are benefits and limitations of those processes.

Our work has shown that despite the significant number of federated tools, algprithms and platforms, when you examine 
the patterns for how these analyses are undertaken they can be simply grouped based on two criteria, 

| Component         | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              | 
|-------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Analytical**    | There are three key concpets that summarise the vast number of analytics and learning that are possible when data is spread across multiple TREs:<br/><br/> **Isolated** – these are analyses that are replicated individually within each TRE. <br/><br/>**Connected** – these are analyses which require multiple rounds of local calculation and aggregation, requiring the TREs to receive results from other TREs to be included in the local calculations. <br/><br/>**Centralised** – these analyses require data to be pooled, even temporarily, for the analyses to be performed.                                               |
| **Data Movement** | As deteremined by the algorithm type, this is how data is required to moves in and out of a TRE and how it is exectued. <br/><br/>**Summary:** where a statistical result is shared back that does not relate to any individual but is a calculation based on the confidential data. It is the summary result that moves from the TRE. <br/><br/>**Model Parameters:** where a machine learning algorithm has been run on the data and the weights are moved from the TRE. <br/><br/>**Row level data**:where row level data is moving, whether that is anonymised (or not) but each data element is that at the level of an individual. |


## Connections
The patterns we have identified to date indicate that the type of analysis to be run dictates the type of data to move. It is important to state that this classifications of the data movement does not indicate the disclosive data of a specific result. Clearly summary results are less liekly to be disclosive than row level data, however, we are not stating any safety element to the data movement. 
| Analytical | Data Movement | 
|-|-|
| **Isolated** | Summary only| 
| **Connected** | Model Parameters| 
| **Centralised**| Row level data only|
